# Privy Multi-Role Profile and Chat Persistence Design

## Goal

Make Privy the authoritative user identity for Shire, persist candidate and
recruiter profiles in hosted Postgres, and isolate assistant history by authenticated
user, active role, and page resource.

This removes the current hard-coded `candidate-001` and `recruiter-001`
identities from production chat requests.

## Scope

This feature includes:

- Server verification of Privy access tokens for profile and chat routes.
- One stable Shire user per Privy user ID.
- Independent candidate and recruiter profiles for the same user.
- Role activation only after the corresponding profile is saved.
- Server-generated chat scope, thread, and memory resource identifiers.
- Trusted profile context for identity-aware assistant responses.
- Drizzle schema, migrations, and server-only Postgres data access.
- Migration of candidate and recruiter form persistence from Zustand to APIs.

This feature does not migrate jobs, applications, stakes, disputes, or
on-chain registry state from the demo store.

## Identity Model

The stable external identity is the verified Privy user ID, for example:

```text
did:privy:abc123
```

The same Privy identity can own both Shire roles:

```text
did:privy:abc123 + candidate
did:privy:abc123 + recruiter
```

Changing role does not create a new login session or Shire user. It only
selects a different role profile and chat namespace.

The browser may request a role, but it cannot assert a user ID. Every protected
route derives the user ID from a server-verified Privy access token.

## Role Activation

A role is active only when its profile has been saved successfully:

- Saving a candidate profile activates `candidate`.
- Saving a recruiter profile activates `recruiter`.
- One user may activate both roles.
- Requests for a role without a saved profile return `403 role-not-active`.

The database is the source of truth for active roles. Client Zustand state is
only a cache for rendering and demo-only data.

## Database Schema

Hosted Postgres (currently Neon) stores three profile-domain tables. Drizzle ORM is
the source of truth for the TypeScript schema, SQL migrations, and server
queries.

### `app_users`

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `uuid` | Primary key generated by Postgres |
| `privy_user_id` | `text` | Unique, immutable external identity |
| `created_at` | `timestamptz` | Defaults to `now()` |
| `updated_at` | `timestamptz` | Maintained on writes |

### `candidate_profiles`

| Column | Type | Notes |
| --- | --- | --- |
| `user_id` | `uuid` | Primary key and FK to `app_users.id` |
| `profile` | `jsonb` | Validated `CandidateProfile` payload |
| `created_at` | `timestamptz` | Defaults to `now()` |
| `updated_at` | `timestamptz` | Maintained on writes |

### `recruiter_profiles`

| Column | Type | Notes |
| --- | --- | --- |
| `user_id` | `uuid` | Primary key and FK to `app_users.id` |
| `profile` | `jsonb` | Validated `RecruiterProfile` payload |
| `created_at` | `timestamptz` | Defaults to `now()` |
| `updated_at` | `timestamptz` | Maintained on writes |

Using one row per user and role enforces role uniqueness while allowing one
user to own both profiles. JSONB keeps the initial migration aligned with the
existing validated TypeScript profile contracts. Frequently queried business
fields can be promoted to typed columns when server-side job matching is
migrated.

Indexes:

- Unique B-tree index on `app_users.privy_user_id`.
- Primary-key indexes cover profile lookup by `user_id`.

All tables have RLS enabled. No `anon` or `authenticated` browser policies are
created because Shire does not use Supabase Auth. Next.js server routes connect
directly to Postgres through Drizzle after Privy authorization. Database URLs
are server-only and are never exposed through a `NEXT_PUBLIC_` variable.

Runtime queries use the provider's pooled URL with prepared statements
disabled. Drizzle Kit migrations use a direct/unpooled URL so schema
changes are not run through transaction-pool limitations.

## Server Authentication

A shared server helper:

1. Reads `Authorization: Bearer <Privy access token>`.
2. Verifies the token with `@privy-io/node`.
3. Extracts the stable Privy user ID.
4. Upserts or resolves the corresponding `app_users` row.
5. Returns a typed authenticated Shire identity.

Missing, expired, malformed, or invalid tokens return `401 unauthorized`.

Demo fallback is allowed only when Privy server configuration is entirely
absent. When Privy is configured, protected routes never fall back to seed
identities.

## Profile APIs

Candidate:

```http
GET /api/profiles/candidate
PUT /api/profiles/candidate
```

Recruiter:

```http
GET /api/profiles/recruiter
PUT /api/profiles/recruiter
```

`PUT` validates the body with the existing Zod schema before upserting it. A
successful upsert activates the role. `GET` returns `404 profile-not-found`
when onboarding for that role has not been completed.

Candidate CV upload continues to resolve the same authenticated Privy user.
The generated CV draft remains pending review; only `PUT
/api/profiles/candidate` persists the confirmed profile.

## Client Profile Flow

The profile forms receive their initial values from the profile APIs.

On save:

1. The client obtains a Privy access token.
2. It sends the validated profile to the role-specific API.
3. The server verifies Privy and persists the profile.
4. The client updates its Zustand cache after server success.

On role switch, the app loads the selected role profile. A role without a
profile redirects to that role's onboarding flow instead of creating a fake
identity.

## Chat Request Flow

The browser sends only:

- Requested role.
- Current resource type and resource ID, when applicable.
- Current messages.
- Privy bearer token.

The browser does not control:

- `viewerId`
- `threadId`
- `resourceKey`
- memory identifiers
- trusted profile context
- system identity instructions

The Next.js chat route:

1. Verifies the Privy token.
2. Resolves the Shire user and requested role profile.
3. Rejects inactive roles with `403`.
4. Validates access to the requested resource.
5. Builds the trusted scope and memory identifiers.
6. Adds a bounded, trusted role profile context.
7. Forwards the request to the agent using the internal service token.

The agent must reject direct chat requests without the internal service token.

## Mastra Memory Keys

Mastra distinguishes conversation history from the owner of that history:

```text
resource = user:<shire-user-uuid>:role:<role>
thread   = user:<shire-user-uuid>:role:<role>:<scope>
```

Examples:

```text
resource:
user:9a60c8e0-2e7c-4f04-b536-63ecb68b7c19:role:candidate

candidate general:
user:9a60c8e0-2e7c-4f04-b536-63ecb68b7c19:role:candidate:general

candidate job:
user:9a60c8e0-2e7c-4f04-b536-63ecb68b7c19:role:candidate:job:job-001

recruiter general:
user:9a60c8e0-2e7c-4f04-b536-63ecb68b7c19:role:recruiter:general
```

The stable user UUID remains the same across roles. Role and resource segments
isolate message history. The role-specific resource is intentionally stricter
than sharing one resource across roles because Mastra working memory and
semantic recall currently use resource scope. Stable identity remains
represented by the same user UUID, while learned facts and preferences remain
role-specific.

## Trusted Assistant Context

The server provides a compact context object derived from the saved role
profile. Candidate context may include:

- Display name.
- Bio.
- Skills.
- Target roles.
- Experience level.
- Location and timezone.

Recruiter context may include:

- Company name.
- Company description.
- Website.
- Location.

The context is marked as trusted application data. Raw access tokens, Privy
claims, full CV documents, secrets, and unrelated profile fields are excluded.

For a saved candidate named `Alex Bizzer`, asking "nama aku siapa?"
should be answerable from this trusted context. Without a saved profile, the
role is inactive and the chat request is rejected rather than guessed.

## Security Rules

- Never trust identity or memory keys supplied by the browser.
- Never expose database URLs or the agent service token to the browser.
- Verify Privy tokens on every protected profile, CV, and chat request.
- Validate requested role against persisted profiles.
- Validate all profile writes with Zod.
- Keep Supabase tables RLS-enabled with no public policies.
- Bound profile context size before forwarding it to the agent.
- Do not log access tokens, Supabase keys, or full profile payloads.
- Agent chat endpoints require the internal service token.

## Error Handling

Stable API errors:

- `401 unauthorized`: missing or invalid Privy token.
- `403 role-not-active`: requested role profile does not exist.
- `403 resource-forbidden`: role cannot access the requested resource.
- `404 profile-not-found`: role onboarding has not been completed.
- `400 invalid-profile`: profile payload failed schema validation.
- `500 missing-database-configuration`: required Postgres URL absent.
- `502 agent-unreachable`: authenticated request could not reach the agent.

The UI should direct `role-not-active` and `profile-not-found` users to the
matching onboarding page.

## Migration and Compatibility

Existing localStorage profiles are not silently trusted or uploaded. After
this feature:

- Authenticated users load profiles from Supabase.
- A missing server profile triggers onboarding.
- Demo mode may continue using Zustand when Privy is not configured.
- Seed jobs and other demo records remain available.

Existing Mastra chat rows using seed thread IDs remain in the local LibSQL
database but are not attached to authenticated users. New authenticated keys
start clean, correctly isolated histories.

## Testing

Unit and route tests cover:

- Privy user ID cannot be overridden by browser payload.
- Candidate and recruiter profiles can coexist for one user.
- Unsaved roles are rejected.
- Profile writes are schema-validated.
- Chat scope and memory keys are rebuilt server-side.
- Candidate and recruiter threads differ for the same user.
- Two Privy users never receive the same memory keys.
- Trusted profile context contains the saved name.
- Invalid tokens return `401`.
- Agent chat requires the internal service token.

Database verification covers:

- Migration applies cleanly.
- RLS is enabled.
- Public roles cannot read or write profile tables.
- Drizzle can transact, upsert, and retrieve both role profiles.

An end-to-end manual check uses two Privy accounts and both roles to confirm
that names and chat histories do not cross user or role boundaries.
